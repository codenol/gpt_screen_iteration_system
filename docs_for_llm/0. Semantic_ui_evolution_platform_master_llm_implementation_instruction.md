# Semantic UI Evolution Platform
# Master LLM Implementation Instruction

# 1. Mission

You are implementing a platform called:

```text
Semantic UI Evolution Platform
```

This is NOT:
- a no-code builder
- a drag-and-drop editor
- a frontend code generator
- a production React generator
- a Figma replacement

This IS:

```text
collaborative semantic UX evolution environment
```

The platform exists to:
- generate interactive semantic screens
- allow collaborative UX review
- evolve screens through semantic feedback
- preserve UX history
- provide explainable UI evolution
- allow iterative product/design collaboration

The system is:
- semantic-first
- collaboration-first
- evolution-first
- review-first
- deterministic
- explainable
- constrained

---

# 2. Core Philosophy

The platform NEVER works with:
- arbitrary JSX
- arbitrary React trees
- arbitrary frontend logic
- arbitrary layouts
- arbitrary interactions

The platform works with:

```text
semantic UI representation
```

UI is treated as:
- semantic structure
- UX artifact
- composition of capabilities
- chain of semantic transformations
- evolving design object

---

# 3. Critical Architectural Rule

The platform MUST NOT be centered around LLM.

The platform MUST be centered around:
- Design Protocol
- Semantic Nodes
- Widget System
- Revision System
- Transform Engine
- Renderer

LLM is ONLY:

```text
semantic reasoning layer
```

This is mandatory.

# 6. Design Protocol System

Design Protocol System (ДС) is the main intelligence layer.

ДС defines:
- UX grammar
- widget semantics
- interaction contracts
- archetypes
- layout semantics
- UX heuristics
- transform rules
- semantic compatibility
- allowed patterns
- discouraged patterns

ДС is:

```text
UI grammar + UX knowledge base + transform rules engine
```

Without ДС the platform becomes unstable.

- semantic_ui_evolution_platform_section_3_design_protocol.md

# 7. Widget System

The platform uses ONLY predefined widgets.

Widgets are NOT plain React components.

Widgets are:

```text
semantic interactive capabilities
```

Every widget MUST contain:
- semantic metadata
- built-in interactions
- variants
- transform capabilities
- semantic nodes
- renderer bindings
- explainability metadata

Widgets MUST NOT require manual interaction scripting.

Interactions are built-in.

Examples:
- DataTable
- FiltersBar
- Form
- Drawer
- Timeline
- Chart
- StatsCards
- ActivityFeed

Reference:
- semantic_ui_evolution_platform_section_10_widget_system.md

---

# 8. Missing Capability Widget

If required capability does not exist:
- renderer MUST show placeholder widget
- placeholder MUST be visually distinct
- placeholder MUST preserve semantic intent

This is NOT an error.

This is platform evolution mechanism.

# 9. Semantic Node System

Semantic Node System is foundational.

Everything MUST be built around semantic nodes.

Semantic nodes are:
- review targets
- transform targets
- comment targets
- explainability anchors
- revision anchors

Semantic nodes are NOT:
- DOM nodes
- React components
- CSS selectors

Every semantic node MUST have:
- stable id
- semantic role
- visual mapping
- hierarchy path
- revision continuity

Example:

```text
users_table.filters.status
```

Renderer MUST support:
- visual-to-semantic mapping
- semantic-to-visual mapping
- granular selection
- nested targeting

Reference:
- semantic_ui_evolution_platform_section_9_semantic_node_system.md

# 11. Revision System

Revision System MUST use event-sourced philosophy.

Source of truth is:

```text
semantic transform history
```

NOT snapshots.

Current state:

```text
Initial State
 + Transform Chain
 = Current Revision
```

Revisions MUST be:
- immutable
- explainable
- reviewable
- branchable
- comparable

Primary UX for revisions:
- visual comparison
- revision switching
- applied comments visualization

NOT git diffs.

- revision compare
- branch switching

# 15. Transform System

Transforms are semantic UX operations.

Examples:
- increase_density
- simplify_filters
- collapse_secondary
- move_to_sidebar
- prioritize_search

Transforms MUST be:
- deterministic
- composable
- explainable
- reversible
- validatable

Transforms MUST NOT directly mutate renderer state.

Transforms create:

```text
new revision
```
